01 · Principi guida

Cosa resta dentro, cosa resta fuori

Lo scenario Hobby project personale pone confini chiari: PloppyTV resta una PWA local-first, gratuita, open source, manutenuta da te. Le feature vanno scelte in base a "lo userò davvero?" e non a "servirebbe per monetizzare?".

✓

Local-first sempre

Tutti i dati su localStorage + JSON export. Nessun backend, nessun account, nessun sync server. Multi-device via export/import manuale o via sync client-side (es. Google Drive Picker API). Backup automatico opzionale su cloud storage dell'utente.

✓

Privacy by design

Nessun tracking, nessun analytics, nessun cookie di terze parti. Le uniche chiamate network sono verso TVMaze (metadata) e, opzionalmente, verso un LLM per AI discovery (con toggle esplicito e API key inserita dall'utente). GDPR native senza sforzo.

✓

PWA cross-platform

Una codebase, tutte le piattaforme: iOS, Android, desktop, tablet. Installabile 1-click, funziona offline, aggiornamenti via service worker. zero fee Apple/Google. Distribuzione ai conoscenti = "vai su questo URL e premi Aggiungi a Home".

✓

Open source MIT

Repo pubblica, audit-able, fork-able. LICENSE file MIT aggiunto in P1. I conoscenti possono ispezionare il codice, proporti modifiche, forkare per personalizzazioni. Community potenziale ma non obbligo.

✗

Niente backend

Nessun server da mantenere, pagare, monitorare. Nessun database. Nessuna API REST. Se serve sync multi-device, si usa cloud storage esistente (Google Drive, iCloud, Dropbox) via OAuth client-side — mai un backend proprietario.

✗

Niente account / login

L'utente apre l'app e inizia a usarla. Nessuna registrazione, nessuna email, nessuna password. Per sync opzionale tra dispositivi, si usa l'account Google/Apple che l'utente ha già — mai un sistema di auth custom.

✗

Niente monetizzazione

Nessun paywall, nessun abbonamento, nessun ads. Se in futuro vuoi accettare donazioni spontanee, GitHub Sponsors o "Buy me a coffee" link nell'about. Ma niente feature locked dietro pagamento.

✗

Niente social community

Nessun follower, nessuna bacheca pubblica, nessun sistema di recensioni condivise. Le recensioni e i rating sono privati — solo tu vedi le tue opinioni. Posizione coerente con "hobby personale" e anti-burnout.

Test decisionale: prima di aggiungere qualsiasi feature, chiediti "lo userò io almeno una volta a settimana?" e "lo spiegherei in 30 secondi a un amico non tecnico?". Se una delle due risposte è no, la feature non entra in roadmap. Questo filtro mantiene il progetto leggero e curato.

03 · Anti-roadmap

Cosa NON fare (anti-pattern hobby)

Feature che appaiono nella roadmap "indie premium" ma che NON devono entrare in un progetto hobby personale. Riconoscerle è altrettanto importante quanto pianificare cosa fare.

✗

Backend proprietario

Nessun Supabase, Firebase, Vercel functions, Postgres. Se serve sync, usa cloud storage esistente (P3). Un backend = costi, monitoring, security, downtime, burnout. TV Time è morta anche per questo.

✗

Account system custom

Nessuna registrazione email/password, nessun OAuth custom per "account PloppyTV". Se serve identità per sync, usa l'account Google/Apple che l'utente ha già. Mai gestire password.

✗

API pubblica OAuth2

Nessuna API REST esposta. È il moat di Trakt, ma richiede infrastruttura, rate limiting, auth, docs, versioning. Per uso hobby non serve: l'utente usa la PWA direttamente.

✗

Paywall / premium tier

Nessuna feature locked dietro abbonamento. Se vuoi donazioni spontanee, metti un link "Support the project" nell'about (GitHub Sponsors / Buy me a coffee). Ma niente gating.

✗

Social community features

Nessun follower, nessun profilo pubblico, nessuna bacheca, nessun sistema di recensioni condivise. I rating sono privati. Per condividere opinioni con amici usa chat esterne (WhatsApp, Telegram).

✗

Scrobbling Plex/Kodi/Netflix

Richiede integrazioni native, webhook, bridge server. Complessità enorme per ROI nullo in un progetto personale. Se ti serve scrobbling, usa Trakt per quello e PloppyTV come tracker separato.

✗

Analytics / tracking utenti

Nessun Google Analytics, nessun Plausible, nessun PostHog. Se vuoi sapere come si usa l'app, chiedi ai tuoi conoscenti direttamente. Privacy-first è USP, non ostacolo.

✗

Multi-language support espanso

IT + EN (P2) bastano per conoscenti. Aggiungere FR/ES/DE richiede traduzioni mantenute, contributors, CI. Per un progetto hobby è overhead. Se un amico vuole FR, fork e traduce lui.

Red flag da watchare: se ti ritrovi a pensare "questa feature sarebbe figa ma serve un server", fermati. Significa che la feature non è adatta al tuo scenario hobby. Torna ai principi P1-P4: local-first, no backend, no account, no monetizzazione. Il constraint genera creatività, non limita.
